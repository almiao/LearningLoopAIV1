# 对话引擎收敛版目标模型草案

本文档给出当前对话引擎的**收敛版目标模型**，目标不是推翻现有链路，而是把已经存在的概念重新分组、理清边界，并逐步迁移到更稳定的 tutor policy 结构。

原则：

- 不按知识点写规则
- 不把模型锁死成固定模板
- 工程层只做上下文整理、边界保护、展示投影
- 模型负责诊断、决策、生成
- 旧字段保留一段时间，逐步迁移

---

## 1. 目标：四组上下文 + 一组会话投影

当前我们把大量字段平铺在一个大对象里。收敛后，目标是把对话相关信息整理成下面四组：

1. `topic_context`
2. `interaction_context`
3. `teaching_context`
4. `orchestration_context`

另外，面向前端与评测展示，单独保留：

5. `display_projection`

---

## 2. 目标对象模型

### 2.1 `topic_context`

只回答一个问题：

- **现在正在讲什么，当前要答什么，当前范围是什么**

建议字段：

```json
{
  "concept_id": "timeout-retry-idempotency-boundary",
  "concept_title": "超时 / 重试 / 幂等边界",
  "current_probe": "为什么一个接口支持超时重试，不等于它天然就安全？",
  "question_meta": {
    "type": "provenance-backed",
    "label": "系统生成诊断题"
  },
  "scope": {
    "type": "domain",
    "id": "service-reliability",
    "title": "服务可靠性与高可用"
  },
  "target": {
    "id": "bigtech-java-backend",
    "title": "大厂 Java 后端面试包"
  }
}
```

设计说明：

- 这是“当前题真相”的唯一聚合对象
- prompt 应优先围绕它判断当前回合上下文

---

### 2.2 `interaction_context`

只回答一个问题：

- **用户现在是在回答、求讲解、切题，还是已经疲劳**

建议字段：

```json
{
  "explicit_intent": "teach|advance|",
  "burden_signal": "normal|high",
  "interaction_preference": "balanced|probe-heavy|explain-first",
  "engagement": {
    "answer_count": 3,
    "teach_request_count": 1,
    "skip_count": 0,
    "repeated_control_count": 0
  },
  "friction": {
    "answer_length": 42,
    "answer_is_blank": false,
    "fatigue_level": "low"
  },
  "should_slow_down": false
}
```

设计说明：

- 这是现在分散在 `intent / engagement / friction_signals / stop_conditions` 里的交互信息收敛版
- 后续可以把 `stop_conditions` 的一部分保留在 orchestration，用户状态放到这里

---

### 2.3 `teaching_context`

只回答一个问题：

- **这题已经教到哪了，现在还差什么**

建议字段：

```json
{
  "working_diagnosis": {
    "state": "partial",
    "confidence_level": "medium",
    "reasons": ["..."],
    "open_questions": ["..."]
  },
  "anchor_history": {
    "recent_turns": [],
    "teach_count": 1,
    "has_recent_teaching": true,
    "recent_takeaways": ["..."]
  },
  "anchor_summary": {
    "confirmed_understanding": "...",
    "current_gap": "...",
    "last_teaching_point": "...",
    "last_updated_at_turn": 5
  },
  "recent_evidence": [],
  "memory_anchor_summary": {}
}
```

设计说明：

- `working_diagnosis` 是现在 `runtimeMap` 的可读化版本
- `anchor_history` 是原始近场教学历史
- `anchor_summary` 是后续最重要的新增摘要对象

其中：

- `anchor_history` 负责保留原始近场信息
- `anchor_summary` 负责提供“可执行教学摘要”

建议后续逐步把 prompt 的主要历史输入从原始 `recent_turns` 迁到 `anchor_summary`

---

### 2.4 `orchestration_context`

只回答一个问题：

- **工程层现在允许怎么走**

建议字段：

```json
{
  "budget": {
    "max_probe_turns": 3,
    "probe_turns_used": 1,
    "remaining_probe_turns": 2,
    "max_teach_turns": 2,
    "teach_turns_used": 1,
    "remaining_teach_turns": 1
  },
  "stop_conditions": {
    "probe_budget_reached": false,
    "teach_budget_reached": false,
    "friction_high": false,
    "recent_info_gain_level": "medium",
    "should_discourage_more_probe": false
  },
  "latest_control_verdict": null,
  "workspace_scope": {
    "type": "domain",
    "id": "service-reliability"
  }
}
```

设计说明：

- 这组是工程保护层，不是教学内容层
- 它告诉模型“哪些动作目前风险高、预算低、可能需要收口”

---

### 2.5 `display_projection`

只回答一个问题：

- **前端和评测该怎么把这轮投影成用户可读内容**

建议字段：

```json
{
  "summary_candidate": "...",
  "followup_candidate": "...",
  "show_takeaway": true,
  "show_candidate_followup": false
}
```

设计说明：

- 这是展示层对象，不建议参与模型推理
- 它的作用是减轻 UI 对原始工程 turn 的直接依赖

---

## 3. 各对象边界

### `topic_context`

- 只描述“这轮正在处理的题”
- 不放用户疲劳和预算

### `interaction_context`

- 只描述用户状态
- 不放题目知识诊断

### `teaching_context`

- 只描述这题的教学历史、诊断和缺口
- 不放工程 stop 条件

### `orchestration_context`

- 只描述工程层边界、预算、裁决
- 不描述教学内容本身

### `display_projection`

- 只面向展示
- 不面向模型推理

---

## 5. 迁移策略

### Phase 1：新增聚合对象，不删旧字段

做法：

- 在 `context_packet` 中新增：
  - `topic_context`
  - `interaction_context`
  - `teaching_context`
  - `orchestration_context`
- 旧字段继续保留
- prompt 开始优先引用新对象

目标：

- 不破坏现有链路
- 让 review 和 prompt 阅读都更清楚

### Phase 2：引入 `anchor_summary`

做法：

- 每个 concept state 中新增：
  - `confirmed_understanding`
  - `current_gap`
  - `last_teaching_point`
  - `last_updated_at_turn`
- 每轮更新

目标：

- 减少模型反复从 raw turns 中重建“这题已经讲到哪”

### Phase 3：prompt 主要围绕新模型工作

做法：

- turn prompt 不再主要看平铺字段
- 优先读取：
  - `topic_context`
  - `interaction_context`
  - `teaching_context`
  - `orchestration_context`

目标：

- 让模型决策顺序更稳定
- 降低字段重叠带来的歧义

### Phase 4：清理旧字段

前提：

- 前端
- 评测脚本
- AI service
- legacy 对照链路

都已经切换到新模型后，再逐步移除旧 alias。

---

## 6. 当前建议的 prompt 使用顺序

建议在主 turn prompt 中明确：

1. 先看 `topic_context`
2. 再看 `interaction_context`
3. 再看 `teaching_context`
4. 最后看 `orchestration_context`

这样做的意义是：

- 先知道当前题是什么
- 再知道用户现在想干嘛
- 再知道这题已经教到哪
- 最后再考虑工程约束

---

## 7. 为什么这个模型更适合当前引擎

因为当前问题并不是“字段不够”，而是：

- 真相分散
- 边界重叠
- 模型很难知道该优先看什么

收敛后的价值：

- 对模型：上下文优先级更清楚
- 对工程：编排边界更清楚
- 对 review：能快速判断每层职责
- 对后续重构：可以逐步迁移，不需要一次推翻

---

## 8. 本轮建议先推进的实现

建议当前优先做：

1. Phase 1：在 `context_packet` 中正式加入四组聚合对象
2. 主 prompt 改成优先围绕这四组对象组织决策
3. 文档与测试同步

建议暂缓做：

1. 强制删除旧字段
2. 用工程约束替代模型决策
3. 把 `display_projection` 直接塞进模型输入
